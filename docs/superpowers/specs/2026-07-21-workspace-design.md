# WorkSpace — дизайн-спека

> Дата: 2026-07-21. Статус: согласована на брейншторме, ждёт ревью перед планом
> реализации.

## 1. Что это

**WorkSpace** — локальный однопользовательский планировщик задач. Две проекции
одних и тех же задач:

- **Дерево** — структурированный вид: ветки произвольной вложенности
  («Работа», «Быт», «Проект X» → подзадачи → под-подзадачи…).
- **Неделя** — календарная раскладка: 7 колонок Пн–Вс, задачи на своих днях.

Пользователь переключается между видами; задача попадает в день назначением
даты (из дерева или созданием прямо в дне недели).

Происхождение: дизайн (визуальный язык, токены, UI-идиомы) берётся из проекта
`space` (`/Users/aydrus/space`); функциональность — своя, сильно урезанная.
Логика space — только контекст, не источник требований.

### Использование

Локально на рабочем компьютере, запуск через `docker compose up`, данные — в
SQLite-файле на volume. Один пользователь, без auth, без сети наружу.

### Вне scope (осознанные срезы)

- Mobile-вид: использование только за десктопом; вёрстка desktop-first без
  обязательств на узкие viewport'ы. Триггер возврата: появление реального
  сценария «смотреть с телефона».
- Привычки, повторяющиеся задачи, сферы, цели, деньги, входящие, заметки —
  весь остальной функционал space.
- Мультипользовательность, синхронизация, экспорт.
- Приоритеты, оценки, статус «в работе» — задача либо сделана, либо нет.

## 2. Ключевые решения (зафиксированы с пользователем)

| Решение | Выбор |
|---|---|
| Архитектура видов | Два полноэкранных вида с переключателем; в дереве внизу — полоска текущей недели как дроп-зона |
| Дата | У любого узла дерева (и листа, и родителя), одна, опциональная |
| Верхний уровень дерева | Несколько корней (`parent_id IS NULL`) |
| Done родителя | Независим от детей; рядом счётчик детей «3/5» |
| Поля задачи | Минимум: название, описание, done, дата. Без статусов и приоритетов |
| Бэклог-панель в неделе | Нет — раскидывание по дням идёт из дерева |
| Стек | Go + SQLite, один контейнер; фронт React 19 + Vite + Tailwind v4 |
| Темы | Тёмная (дефолт) и светлая, токены из space |

## 3. Модель данных

Одна таблица `tasks`:

| Поле | Тип | Правила |
|---|---|---|
| `id` | INTEGER PRIMARY KEY | autoincrement |
| `parent_id` | INTEGER NULL REFERENCES tasks(id) | NULL = корень; FK с проверкой существования |
| `title` | TEXT NOT NULL | непустой после trim |
| `description` | TEXT NOT NULL DEFAULT '' | |
| `done` | INTEGER NOT NULL DEFAULT 0 | bool; независим от детей |
| `scheduled_on` | TEXT NULL | `YYYY-MM-DD`; NULL = только в дереве |
| `position` | INTEGER NOT NULL | порядок среди сиблингов в дереве |
| `day_position` | INTEGER NULL | порядок внутри дня; NULL когда `scheduled_on` NULL |
| `created_at` | TEXT NOT NULL | RFC 3339 UTC |
| `updated_at` | TEXT NOT NULL | RFC 3339 UTC |

Индексы: `(parent_id)`, `(scheduled_on)`.

### Инварианты

1. **Нет циклов.** Reparent проверяется на сервере: новый родитель не может
   быть потомком переносимого узла (и не самим узлом). Нарушение → 422.
2. **Каскадное удаление.** Удаление узла удаляет всё поддерево одной
   транзакцией. UI перед этим показывает подтверждение с числом задач внутри.
3. **`day_position` живёт вместе с датой.** Снятие даты обнуляет
   `day_position`; назначение даты ставит задачу в конец дня.
4. **Счётчики не хранятся.** «3/5 детей сделано» вычисляется на клиенте.

## 4. API

REST, JSON, префикс `/api`. Однопользовательский локальный сервис — без auth.

| Метод | Путь | Что делает |
|---|---|---|
| `GET` | `/api/tasks` | Все задачи плоским списком; дерево строит клиент |
| `POST` | `/api/tasks` | Создать: `title`, опционально `parent_id`, `scheduled_on`, `description`. Позиция — в конец сиблингов (и в конец дня, если есть дата) |
| `PATCH` | `/api/tasks/{id}` | Частичное обновление любого подмножества: `title`, `description`, `done`, `scheduled_on`, `parent_id`, `position`, `day_position`. Семантика «поле пришло → применяем», null значим (снять дату, сделать корнем) |
| `DELETE` | `/api/tasks/{id}` | Удалить с поддеревом; в ответе — число удалённых |

Ошибки: 404 (нет узла), 422 (валидация: пустой title, битая дата, цикл,
несуществующий parent) — JSON `{error: string}` с внятным текстом.

Личный масштаб (сотни–тысячи задач) → `GET /api/tasks` отдаёт всё разом,
без пагинации. Это решение, а не упущение: одна загрузка держит оба вида
и полоску недели консистентными без серверных выборок по датам.

## 5. Клиент: данные

Паттерн DataProvider из space: один контекст со всеми задачами.

- Старт: `GET /api/tasks` → нормализованный стор (map по id + производные
  индексы: дети по parent_id, задачи по дате).
- Мутации: **оптимистично** — применить локально, отправить запрос; при ошибке
  откатить и показать toast. После успеха — сверка ответом сервера.
- Порядок drag-операций (position / day_position) шлётся одним PATCH
  перетащенной задачи; пересчёт позиций сиблингов — на сервере одной
  транзакцией (перенумерация), клиент забирает актуальные позиции из ответа.

## 6. Вид «Дерево»

- Узлы с шевронами; состояние раскрытия — в localStorage (переживает
  перезапуск).
- Строка узла: чеквбокс done · название · приглушённый счётчик «3/5» (если
  есть дети) · чип даты (если назначена, формат «Вт 22»).
- Клик по чипу даты (или по «＋дата» в hover-действиях) — меню:
  **Сегодня · Завтра · Пн–Вс текущей недели · календарь · Снять дату**.
- Inline-операции: добавить ребёнка, добавить сиблинга, переименовать,
  удалить (с подтверждением при поддереве). Описание — в раскрывающейся
  детали узла.
- Drag узла по дереву — reparent/reorder.
- **Полоска недели** внизу экрана: 7 ячеек (Пн–Вс: имя дня, число, счётчик
  задач дня), стрелки ◂ ▸ листают полоску по неделям, кнопка «Сегодня»
  возвращает. Drop узла на ячейку → назначить дату. Полоска — только
  дроп-зона и индикатор, не список задач.

## 7. Вид «Неделя»

- 7 колонок Пн–Вс; шапка: диапазон («21–27 июля») + «◂ Сегодня ▸».
  Сегодняшний день выделен.
- Карточка задачи в ячейке: чекбокс · название · приглушённая хлебная крошка
  родителя («Проект X / Бэкенд»). Клик — раскрыть деталь (описание,
  переход к узлу в дереве).
- Drag: между днями (смена даты) и внутри дня (day_position).
- Быстрый ввод внизу каждой ячейки: Enter создаёт задачу с датой этого дня,
  **корневым узлом** (перевесить можно потом в дереве).
- Плашка **«Просрочено»** над сеткой: несделанные задачи с датой < сегодня;
  у каждой действия «на сегодня» и «снять дату». Сворачивается; на прошлых
  неделях не показывается.
- Прошлые/будущие недели листаются без ограничений; сделанные задачи остаются
  видимыми на своих днях.

## 8. Переключение видов и оболочка

- Переключатель «Дерево / Неделя» в шапке; роуты `/` (дерево) и `/week`
  (неделя, опционально `/week/YYYY-MM-DD` — любой день недели для листания
  с сохранением позиции в URL).
- Тумблер темы в шапке; тема — `data-theme` на `<html>`, boot-скрипт против
  FOUC (как в space).
- Горячие клавиши: `1`/`2` — переключение видов; `T` — «Сегодня» в неделе.

## 9. Дизайн-система (перенос из space)

Берём как есть, выкидывая неиспользуемое:

- Токены обеих тем из `index.css` (bg/panel/line/text/dim/accent «приглушённое
  золото»/pos/over), радиусы (panel 18 / ctl 12 / box 7), шрифты Onest +

- Идиомы: `.panel`, mono-caps метки (MLabel), чекбоксы (Check — хватает

  (TabBar, CaptureSheet).
- `--over` (просрочка) — единственный алармовый цвет, используется в плашке
  «Просрочено» и на чипах прошедших дат.

## 10. Инфраструктура

- **Один Go-бинарь**: REST + статика фронта через `embed.FS`. Go-версия
  актуальная; утилиты — через `tool` директивы в `go.mod`.
- SQLite: драйвер `modernc.org/sqlite` (CGO-free), `WAL`, `busy_timeout`,
  `foreign_keys=ON`. Файл — `/data/workspace.db`.
- Миграции: версионные, применяются при старте (embedded SQL + таблица
  `schema_migrations`).
- HTTP: таймауты сервера (read/write/idle), graceful shutdown по SIGTERM
  (контекст доводит запросы), логирование `slog` с контекстом.
- Docker: multi-stage (node → сборка фронта; go → бинарь с embed; финальный
  минимальный образ). `docker-compose.yml`: один сервис, порт `8787`,
  volume `./data:/data`.
- Dev-режим без docker: `go run` + `vite dev` с proxy на api (для разработки).

## 11. Ошибки и краевые случаи

- Ошибка мутации → откат оптимистичного состояния + toast с текстом сервера.
- Потеря сервера (компоуз упал) → баннер «нет связи», повтор по кнопке.
- Защита от двойного submit в quick-add (disabled на время запроса).
- Пустые состояния: пустое дерево («создай первую ветку»), пустая неделя,
  пустая ячейка дня.
- Конкурентных клиентов не предполагаем (один пользователь), но сервер всё
  равно корректен: все многошаговые мутации (reparent + перенумерация,
  каскадное удаление) — в транзакциях.

## 12. Тестирование

- **Бэкенд**: unit-тесты сервисного слоя — защита от циклов, каскадное
  удаление, reparent с перенумерацией, назначение/снятие даты и
  `day_position`, валидация. `httptest` на ручки (коды, JSON-контракт).
- **Фронт**: `tsc --noEmit` + lint; логика построения дерева и недельных
  индексов — unit (vitest).
- **Smoke в браузере** перед закрытием: создать ветку → вложить подзадачи →
  раскидать по дням (меню даты + drag на полоску) → перейти в неделю →
  drag между днями → быстрый ввод в день → отметить сделанное → плашка
  просрочки → переключить тему.

## 13. Acceptance criteria

1. `docker compose up` поднимает приложение на `localhost:8787`; данные
   переживают перезапуск контейнера (volume).
2. В дереве: создание корней и вложенных задач, переименование, описание,
   отметка done (независимая), счётчик детей, каскадное удаление с
   подтверждением, drag-перенос по дереву, назначение даты через меню и
   drag на полоску недели.
3. В неделе: задачи на своих днях с крошкой родителя, отметка done, drag
   между днями и внутри дня, быстрый ввод в ячейке, плашка просрочки с
   «на сегодня»/«снять дату», листание недель, выделенный «сегодня».
4. Переключение видов мгновенное, состояние (раскрытие дерева, текущая
   неделя) не теряется в рамках сессии.
5. Обе темы работают на всех экранах; тёмная — дефолт.
6. Тесты бэкенда и typecheck фронта зелёные; smoke-прогон пройден.
