# WorkSpace v2 — проекты, цвета, каскад done

> Дата: 2026-07-21. Дельта к базовой спеке `2026-07-21-workspace-design.md`.
> Запрос пользователя: убрать полоску недели из дерева; раздел «Дерево» →
> «Проекты» с сайдбаром проектов слева и деревом выбранного проекта справа;
> цвет у проекта + тонкая цветная полоска у задач в дереве и в неделе;
> авто-каскад done вверх по родителям.

## 1. Изменения против v1

| Было (v1) | Стало (v2) |
|---|---|
| Один общий лес: корни дерева = верхнеуровневые ветки | Сущность **Проект** (имя, цвет, порядок); у каждого проекта свой лес задач |
| Раздел «Дерево», роут `/` | Раздел «Проекты», роут `/projects/:id?`; `/` редиректит |
| Полоска недели внизу дерева (дроп-зона) | **Удалена** — назначение дат только через меню даты; граница «структура/планирование» чище |
| Done родителя независим | **Авто-каскад** (правила в §4); ручная отметка по-прежнему возможна |
| Quick-add в неделе создаёт корневую задачу «в никуда» | Quick-add создаёт корневую задачу **в выбранном проекте** (селектор-точка в ячейке) |

## 2. Модель данных

Новая таблица `projects`:

| Поле | Тип | Правила |
|---|---|---|
| `id` | INTEGER PK | |
| `name` | TEXT NOT NULL | непустой после trim |
| `color` | TEXT NOT NULL | `#rrggbb`; выбирается из клиентской палитры, сервер валидирует формат |
| `position` | INTEGER NOT NULL | порядок в сайдбаре |
| `created_at`, `updated_at` | TEXT NOT NULL | RFC 3339 UTC |

`tasks` получает `project_id INTEGER NOT NULL REFERENCES projects(id)` —
**у каждой задачи**, не только у корней (денормализация: неделя красит
карточки без прогулок по предкам). Инвариант: `project_id` задачи ==
`project_id` её родителя; поддерживается сервером:

- create: ребёнок наследует проект родителя; корень обязан указать проект;
- reparent под родителя другого проекта: всё поддерево получает проект
  нового родителя (UI межпроектный drag не даёт, но сервер корректен).

### Миграция 0002 (текущие данные)

Каждая корневая задача v1 становится **проектом** (имя = title, цвета —
по кругу палитры), её дети поднимаются до корней этого проекта. Если у
корня были значимые поля (дата / done / описание) — внутри нового проекта
создаётся задача-копия с этими полями, чтобы ничего не потерять (пример:
корень «Быт» с датой Чт 23 → проект «Быт» + задача «Быт» на Чт 23).
Пустая база → миграция ничего не создаёт; первый проект создаёт пользователь.

## 3. Палитра

12 фиксированных цветов на клиенте (константа `PALETTE`), первый — акцент
`#c9a96a`; далее зелёный/синий/терракота/фиолет/бирюза/роза и т.д. — все
подобраны читаемыми в обеих темах как **маркёры** (полоса/точка), заливок
нет (правило языка space). Новый проект получает следующий цвет по кругу
(count % 12). Смена цвета — пикер из этих же 12.

## 4. Каскад done

Правила (сервер, в транзакции мутации; все изменённые задачи — в `tasks[]`
ответа):

1. **Отметил задачу done** → вверх по предкам: если у родителя все прямые
   дети done — родитель становится done; продолжаем вверх, пока условие
   выполняется.
2. **Снял done** → все предки теряют done (вверх до корня).
3. **Создал задачу** → все предки новой задачи теряют done.
4. **Перенёс задачу (reparent)**: если в переносимом поддереве есть
   несделанные — новые предки теряют done. Старая ветка **не** авто-
   закрывается, удаление тоже не триггерит авто-done — авто-завершение
   происходит только от явной отметки (правило 1); иначе «удалил задачу —
   проект сам закрылся» станет сюрпризом.

Ручная отметка родителя при несделанных детях остаётся возможной (как в v1);
каскад — надстройка над явными действиями, не запрет.

## 5. API

Новое:

| Метод | Путь | Контракт |
|---|---|---|
| `GET` | `/api/projects` | `{projects:[{id,name,color,position,createdAt,updatedAt}]}` |
| `POST` | `/api/projects` | `{name, color}` → 201 `{project}` |
| `PATCH` | `/api/projects/{id}` | подмножество `{name, color, position}` → `{project}` |
| `DELETE` | `/api/projects/{id}` | каскадом удаляет все задачи проекта → `{deleted: N}` (N — задач) |

Изменения:

- `POST /api/tasks`: принимает `projectId` (обязателен без `parentId`);
  ответ теперь `201 {task, tasks}` — `task` созданная, `tasks` все
  затронутые (созданная + предки, потерявшие done по правилу 3).
- `PATCH /api/tasks/{id}`: без изменений формы (`{tasks}`), но в `tasks`
  теперь попадают и задачи, изменённые каскадом done / сменой проекта.
- Валидации: несуществующий `projectId` → 422; цвет не `#rrggbb` → 422;
  удаление/патч несуществующего проекта → 404.

## 6. UI «Проекты» — все сценарии

Лэйаут: слева **сайдбар проектов** (~230px, sticky), справа — дерево
выбранного проекта. Роут `/projects/:id`; `/projects` и `/` редиректят на
последний открытый (localStorage `workspace-last-project`), иначе на первый
по position, иначе — экран «нет проектов».

Сайдбар, строка проекта: полоска цвета (sbar 3px) · имя · mmeta-счётчик
несделанных задач (пусто, если 0). Активный — фон asoft. Сценарии:

- **Создать**: инпут «＋ Новый проект…» внизу списка; Enter — создать
  (цвет по кругу палитры), проект сразу выбирается, фокус уходит в инпут
  новой задачи справа.
- **Переименовать**: клик по имени активного проекта — inline-инпут
  (Enter/blur — сохранить, Escape — отмена).
- **Цвет**: клик по полоске/точке цвета — поповер-палитра 12 цветов,
  текущий отмечен; выбор применяется сразу.
- **Удалить**: hover ✕ → confirm «Удалить проект „X“ и все его задачи
  (N)?»; после удаления выбирается соседний проект.
- **Порядок**: drag строк сайдбара вертикально (position).
- **Переключение**: клик; выбранный пишется в localStorage; hotkey `1`
  открывает раздел.

Правая панель — дерево как в v1, изменения:

- У **каждой строки** задачи слева тонкая полоска цвета проекта (3×18px,
  как sbar в space) — постоянный маркёр принадлежности.
- Полоска недели внизу удалена; назначение дат — чип/◷ → меню даты.
- Пустой проект: «В проекте пусто. Добавь первую задачу.» + инпут.
- Переход из недели «в дереве →» ведёт на `/projects/:pid?focus=taskId`
  (раскрытие пути, скролл, подсветка — как в v1).
- Каскад: отметил последнего ребёнка → родители закрываются на глазах
  (одна мутация, ответ сервера вливается в стор); добавил задачу в
  закрытую ветку → предки открываются.

## 7. Неделя — изменения

- Карточка задачи: слева тонкая цветная полоска проекта (border-left 3px
  внутри скругления карточки). В плашке «Просрочено» — та же полоска у
  строк.
- **Quick-add**: слева от инпута — кнопка-точка цвета «текущего проекта
  для быстрых задач»; клик — поповер со списком проектов (точка + имя).
  Выбор запоминается (localStorage `workspace-quick-project`); дефолт —
  последний открытый проект. Enter создаёт корневую задачу выбранного
  проекта с датой дня.
- Если проектов нет вообще — quick-add скрыт, вместо него подсказка
  «Сначала создай проект».

## 8. Вне scope v2 (зафиксированные срезы)

- Межпроектный перенос задач в UI (drag дерева работает внутри проекта;
  сервер перенос между проектами уже поддерживает). Триггер возврата —
  реальная потребность перекинуть ветку между проектами.
- Фильтр недели по проекту. Триггер — когда проектов станет много и неделя
  зашумится.

## 9. Тестирование

- Store: projects CRUD, наследование project_id, перекраска поддерева при
  reparent, каскад done по всем четырём правилам, миграция 0002 (корни →
  проекты, копия значимых полей), удаление проекта каскадом.
- API: httptest на новые ручки + новые контракты ответов.
- Smoke в Chrome: создать проекты, цвета, переключение, rename, каскад
  done в обе стороны, полоски в дереве и неделе, quick-add с выбором
  проекта, удаление проекта, обе темы, персистентность после рестарта.
